## 1. 需求

### 背景
进程检测修复已经通过手动验证，当前“检测到真实 CLI 进程数量”的主问题已解决。收尾阶段需要再做一轮更严格的边界检查，确认没有遗漏的前后端一致性问题；如果发现仅影响体验但不涉及策略变更的小问题，应在本轮顺手优化并完成归档。

### 目标
- 对进程检测修复做更严格的前后端链路检查和构建复验。
- 修正收尾检查中发现的轻量体验问题，但不改变“当前 CLI 会话不阻断切换”的既定策略。
- 如无新的阻断问题，完成知识库与归档记录闭环。

### 约束条件
```yaml
时间约束: 作为当前修复后的收尾检查，在本轮内完成
性能约束: 不增加新的高频轮询或重量级后台任务
兼容性约束: 不修改既有进程检测结构的核心语义，不回退已验证通过的显示修复
业务约束: 保持一次只稳定使用一个账号的当前策略，不新增并发切换能力
```

### 验收标准
- [ ] 手动刷新入口能同时刷新工作区状态与进程检测结果，不再依赖轮询才能看到最新进程状态。
- [ ] 若存在会真正阻止切换的进程，按钮/提示文案表达与实际阻断语义一致，不再误导为“所有 Codex 进程都必须关闭”。
- [ ] `bun run build`、`cargo test --manifest-path src-tauri/Cargo.toml` 和本地 `bun run tauri:build` 通过，且未发现新的阻断问题。

## 2. 方案

### 技术方案
在现有进程检测修复基础上，只做两类轻量收尾优化：一是让手动刷新流程显式串联 `check_codex_processes()`，使用户操作后立刻看到最新进程状态；二是把切换禁用提示文案改为“存在阻断切换的 Codex 进程”这类更准确的表达，避免与“检测到 CLI 进程但仍允许切换”的新语义冲突。随后执行一次更贴近交付的完整验证，包括前端构建、Rust 测试与本地 Tauri build。

### 影响范围
```yaml
涉及模块:
  - frontend-app: 刷新动作与切换提示文案优化
  - tauri-commands: 仅复验，无接口语义变更
预计变更文件: 2-4
```

### 风险评估
| 风险 | 等级 | 应对 |
|------|------|------|
| 手动刷新串联进程检测后产生额外一次轻量查询 | 低 | 仅在用户主动点击时触发，不改变轮询频率 |
| 文案调整与现有按钮状态不一致 | 低 | 统一从 `blocking_count/can_switch` 推导提示文本 |

## 4. 核心场景

> 执行完成后同步到对应模块文档

### 场景: 手动刷新后立即看到最新进程状态
**模块**: frontend-app
**条件**: 用户点击“刷新全部”或“刷新工作区状态”
**行为**: 前端同时刷新 usage、workspace auth 和 processInfo
**结果**: 进程数量徽标立即反映最新状态，无需等待轮询

### 场景: 阻断提示文案与实际策略一致
**模块**: frontend-app
**条件**: 存在 `blocking_count > 0` 的阻断型 Codex 进程
**行为**: 切换按钮与悬浮提示明确说明是“存在阻断切换的进程”
**结果**: 用户不会误解为所有 CLI 会话都必须先关闭

## 5. 技术决策

> 本方案涉及的技术决策，归档后成为决策的唯一完整记录

### post-fix-hardening-review#D001: 收尾阶段仅做轻量体验优化，不调整已验证策略
**日期**: 2026-04-14
**状态**: ✅采纳
**背景**: 主问题已经解决且用户已手动确认“完全正常”。当前需要的是更严格的收尾检查和必要的小修，而不是再次改变进程检测策略或切换语义。
**选项分析**:
| 选项 | 优点 | 缺点 |
|------|------|------|
| A: 不再改动，直接归档 | 工作量最小 | 会保留手动刷新不联动进程状态、提示文案不准确这类尾巴 |
| B: 只补轻量体验优化并复验 | 保持策略稳定，同时把边角体验收干净 | 需要追加一次小规模修改与验证 |
**决策**: 选择方案 B
**理由**: 这轮是加强检查，目标本来就是在不扰动主逻辑的前提下把边界问题收干净。B 的收益明确，风险很低。
**影响**: 影响前端刷新链路与切换按钮提示文案；知识库需新增本轮收尾记录
